# Pedals Protocol Specification v1

Pedals is a remote terminal system: an iOS client attaches to PTY sessions hosted by a
desktop daemon, relayed through a zero-knowledge WebSocket relay with end-to-end encryption.

```
[iOS app] <--WSS--> [Relay (Cloud Run)] <--WSS--> [Desktop daemon]
              ciphertext only, relay cannot decrypt
```

## 1. Relay

- Endpoint: `GET /v1/room/:roomId` (WebSocket upgrade), query `role=host|client`.
- `roomId`: 32 hex chars (128-bit random), generated by the desktop daemon. Knowing the
  roomId is the only capability needed to join; security comes from E2EE, not the relay.
- Exactly one `host` per room. v1 allows one `client` (later: N). A new host connection
  replaces the previous one; a new client connection replaces the previous one.
- The relay forwards every binary WebSocket message verbatim to the peer role. It never
  inspects payloads, never persists them.
- When the peer is absent, messages are dropped (not queued). Both sides handle resync
  via `attach`/replay (see §4).
- `GET /healthz` → `200 ok`. Text frames are ignored. Ping/pong every 30 s from relay.
- Env: `PORT` (Cloud Run provides). No other config. No logs of payloads.

## 2. Pairing

The desktop daemon generates: `roomId` (16 random bytes, hex) and `secret` (32 random
bytes). It renders a QR code containing:

```
pedals://pair?v=1&relay=wss%3A%2F%2F<relay-host>&room=<roomId>&s=<base64url(secret)>
```

The iOS app obtains this URL by scanning the QR (device) or pasting it / opening the URL
scheme (simulator: `xcrun simctl openurl booted 'pedals://pair?...'`). Both sides store it
(iOS: Keychain; desktop: `~/.pedals/pairing.json`, mode 0600).

The QR is exchanged physically, so v1 uses a symmetric scheme (no ECDH):

```
key_h2c = HKDF-SHA256(ikm=secret, salt="pedals-v1", info="host->client", 32 bytes)
key_c2h = HKDF-SHA256(ikm=secret, salt="pedals-v1", info="client->host", 32 bytes)
```

## 3. Encryption

Every WebSocket binary message is one encrypted frame:

```
message = seq (u64 LE) || ChaChaPoly.seal(plaintextFrame, key=direction key,
                                          aad=seq bytes).combined
```

- `combined` = CryptoKit combined representation: nonce(12) || ciphertext || tag(16).
  Node side: `chacha20-poly1305` via `crypto.createCipheriv`, random 12-byte nonce.
  (Relay never decrypts; only the two Swift/Node ends of iOS/desktop do. Desktop and iOS
  are both Swift/CryptoKit; Node reference impl exists for tests.)
- `seq` starts at 1 per direction per connection, strictly increasing; receiver drops
  non-increasing seq (replay protection). Fresh WebSocket connection resets seq — each
  side sends `hello` first (see below) carrying a random `connEpoch` to disambiguate.
- Decryption failure → close connection, retry from fresh WS connect.

## 4. Frames (plaintext, after decryption)

```
frame = type (u8) || sessionId (u32 LE) || payload (bytes)
```

| type | name    | dir  | sessionId | payload |
|------|---------|------|-----------|---------|
| 0x00 | ctl     | both | 0         | UTF-8 JSON, see below |
| 0x01 | stdin   | c→h  | sid       | raw bytes to write to PTY |
| 0x02 | stdout  | h→c  | sid       | raw PTY output bytes |
| 0x03 | resize  | c→h  | sid       | cols u16 LE, rows u16 LE |
| 0x04 | replay  | h→c  | sid       | raw bytes: scrollback ring buffer snapshot (send before live stdout after attach) |

### ctl JSON messages

All have `{"t": "<kind>", ...}`:

- `{"t":"hello","who":"host"|"client","connEpoch":<random u32>,"ver":1}` — first frame
  after connect from each side.
- `{"t":"sessions","list":[{"id":1,"title":"zsh — ~/dev","cwd":"/Users/x/dev","rows":40,"cols":120,"createdAt":..., "alive":true}]}` — host→client. Sent on hello, on any change (create/close/title).
- `{"t":"create","cwd":null|"...","cols":120,"rows":40}` — client→host. Host replies with
  updated `sessions` and `{"t":"created","id":N}`.
- `{"t":"close","id":N}` — client→host (also used by CLI locally).
- `{"t":"attach","id":N}` — client→host: subscribe to that session's stdout. Host sends
  `replay` frame (ring buffer, up to 256 KB) then live `stdout`. Client may attach to
  multiple sessions simultaneously (all get stdout; UI shows one).
- `{"t":"detach","id":N}` — stop streaming that session.
- `{"t":"title","id":N,"title":"..."}` — host→client on title change (from OSC 0/2 parsed
  out of PTY output, fallback: foreground process name).
- `{"t":"exit","id":N,"code":0}` — host→client when a session's process exits.
- `{"t":"err","msg":"..."}` — either direction, non-fatal error report.

## 5. Desktop daemon local control (Unix socket)

`~/.pedals/pedals.sock` — newline-delimited JSON-RPC-ish:
request `{"cmd":"ls"|"kill"|"pair"|"status"|"new", "id":N?}`, response
`{"ok":true, ...}` or `{"ok":false,"err":"..."}`.
- `ls` → `{"ok":true,"sessions":[...same shape as ctl sessions...], "client":"connected"|"none","relay":"wss://..."}`
- `kill {id}` → closes PTY.
- `new` → creates a session locally.
- `pair` → `{"ok":true,"url":"pedals://pair?..."}` (also regenerates room+secret if `--reset`).
- `status` → relay connection state, room id, uptime.

CLI subcommands (`pedals ls|kill|new|pair|status`) and the menubar app are both thin
clients of this socket. `pedals serve` runs the daemon in foreground; `pedals pair`
renders the QR as ANSI half-blocks in the terminal plus prints the URL.

## 6. PTY sessions (host side)

- Spawn: `$SHELL` (fallback `/bin/zsh`) as interactive login shell (`-il`), TERM
  `xterm-256color`, COLORTERM `truecolor`, LANG inherited or `en_US.UTF-8`.
- Each session keeps a 256 KB ring buffer of raw output for replay-on-attach.
- Title: parse OSC 0/2 sequences from output stream; fallback to foreground process name
  polled via `tcgetpgrp` every 2 s.
- Default size 120×40 until first `resize`.
